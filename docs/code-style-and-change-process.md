# Стиль кода и процесс внесения изменений

`lampa-source` не навязывает строгий tooling-стиль через ESLint, Prettier или TypeScript. Поэтому основной ориентир здесь - существующий код в конкретной подсистеме.

## Наблюдаемый стиль кода

### JavaScript и модули

- Используются ES modules с `import ... from` и `export default`.
- Большая часть файлов экспортирует один объект, функцию-фабрику или stateful class.
- Проект остается на plain JavaScript, без TypeScript и без схем типов на уровне компиляции.

### Смешение jQuery и native DOM

- jQuery активно используется для поиска элементов, событий и сборки HTML.
- В новых местах встречается и чистый DOM API через `document.createElement`, `Template.js(...)`, `Template.elem(...)`.
- Нормальный стиль проекта - комбинировать оба подхода в одном модуле, если это уже делает окружающий код.

### Событийная модель

- В проекте много event-driven логики: `Subscribe`, `Listener`, `hover:*`, `keydown`, `state:changed`, lifecycle player/full/activity.
- Фокус и навигация завязаны не на обычный click-only UI, а на TV-friendly события вроде `hover:enter`, `hover:focus`, `hover:long`.
- Перед изменением UI почти всегда нужно понять, какой `Controller` и какой lifecycle-сигнал им управляет.

### Именование и комментарии

- Идентификаторы смешанные: часть на английском, часть с русскими комментариями и текстами.
- Комментарии обычно короткие и прикладные.
- Много исторически сложившихся названий с опечатками или устаревшими aliases. Исправлять их только ради красоты не стоит, если это может задеть плагины или сохраненные настройки.

### Форматирование

- Стиль semicolon-light: часть строк заканчивается `;`, часть нет.
- Отступы обычно 4 пробела.
- Простые `function name(){}` встречаются чаще, чем стрелки на верхнем уровне.
- В коде много длинных функций и procedural flow. Если файл уже написан в таком стиле, лучше поддержать его локально, а не вносить резкий stylistic rewrite.

## Как вносить изменения безопасно

### 1. Сначала найти правильный extension point

Перед правкой нужно понять, на каком уровне живет функция:

- экранная навигация - `core/router`, `interaction/activity`, `core/component`;
- шаблоны и HTML - `interaction/template` и `src/templates/*`;
- settings - `interaction/settings/*` или `SettingsApi`;
- data source - `core/api/*`;
- глобальное состояние - `core/storage`, `core/account`, `core/plugins`;
- новые content rows - `core/content_rows`;
- plugin-style расширение - `plugins/*` или внешний plugin API.

Нельзя начинать с косметической правки случайного DOM-узла, пока не понятен владелец поведения.

### 2. Сохранять локальный стиль файла

Практическое правило:

- если файл написан на jQuery, не надо насильно переписывать его целиком на чистый DOM;
- если экран собран через `Lampa.Maker`, лучше продолжать через `params.module`, `params.emit`, `createInstance`;
- если подсистема строится на templates и строковом HTML, лучше продолжать в этом стиле, а не смешивать с новым builder-подходом.

### 3. Обновлять соседние артефакты вместе

Изменение UI почти всегда тянет соседние правки:

- шаблон в `src/templates/*`;
- стили в `src/sass/*` или `plugins/*/css/*.scss`;
- переводы в `src/lang/*`;
- настройки в `SettingsApi` или `settings/*`;
- plugin manifest или `ContentRows`, если меняется точка интеграции.

В проекте много связей по соглашению, а не по типам, поэтому частичная правка быстро ломает runtime.

### 4. Учитывать два UI-подхода

В кодовой базе одновременно живут:

- legacy-компоненты в `src/components/*`;
- модульная система в `src/interaction/*` плюс `Lampa.Maker`.

Если вы меняете старый экран, не нужно автоматически переносить его в новую систему.

Если вы пишете новую изолированную функциональность, лучше смотреть сначала в сторону:

- `Lampa.Maker`
- `Lampa.ContentRows`
- `Lampa.SettingsApi`
- `Lampa.Component.add`

Это уменьшает риск сломать обратную совместимость.

### 5. Не ломать глобальный контракт `window.Lampa`

Плагины и внешние скрипты завязаны на то, что экспортирует `src/app.js`. Поэтому при внутренних изменениях:

- не удаляйте поля из `window.Lampa` без отдельной миграции;
- не переименовывайте исторические API, если на них могут опираться внешние плагины;
- с осторожностью меняйте сигнатуры `Template`, `SettingsApi`, `ContentRows`, `Activity`, `Player`, `Account`.

Даже если модуль внутри ядра кажется "внутренним", он может уже быть частью plugin ecosystem.

### 6. Рекомендованный рабочий процесс

1. Найти владеющую подсистему.
2. Проверить, нет ли уже extension point под нужную задачу.
3. Внести минимальное изменение в правильный слой.
4. Обновить связанные шаблоны, переводы, стили и настройки.
5. Прогнать хотя бы релевантные команды проверки, обычно `npm test` и `npm run build`.
6. Если затронута упаковка plugin/runtime, дополнительно проверить `npm run pack`.

## Где чаще всего правильно менять код

- Новый экран приложения: `src/components/*` или `Lampa.Component.add(...)`, в зависимости от того, расширяете ядро или плагин.
- Новая вариация карточек, линий, категорий: `src/interaction/*` через `Maker` и модульные карты.
- Новый источник данных: `src/core/api/sources/*`.
- Новый ряд на главной: `src/core/content_rows.js` или plugin `Lampa.ContentRows.add(...)`.
- Новый параметр настроек: `Lampa.SettingsApi.addParam(...)`.
- Новый plugin behavior: отдельный `plugins/<name>/<name>.js`, а не врезка в случайный core-файл.

## Антипаттерны для этого проекта

- Массовый stylistic refactor без поведенческой причины.
- Переезд на другой способ рендера в одном изменении.
- Прямой DOM-patch там, где уже есть `Template`, `SettingsApi` или `ContentRows`.
- Удаление deprecated API без проверки plugin-совместимости.
- Переписывание naming-несоответствий вроде `dnla`/`dlna` без понимания последствий для уже опубликованных plugin-файлов и настроек.
